Why nested scrolling needs a height chain

In normal document layout, content creates height. In app layout, the viewport constrains regions. A child like .options can only scroll when its ancestors give it a real bounded height.

.options says overflow: auto, but it has no useful height limit. So it expands, and the outer page/box scrolls instead.

/* Document-style failure */
.options {
  overflow: auto;
}

/* But the ancestors are auto-height.
   The child expands instead of scrolling. */
/* App-shell pattern */
html, body {
  height: 100%;
  margin: 0;
}

body {
  overflow: hidden;
}

.app {
  height: 100%;
  display: grid;
  grid-template-rows: auto 1fr auto;
}

.middle,
aside,
.tools,
.current-tool {
  min-height: 0;
}

.options {
  overflow: auto;
}
The rule of thumb: a scrollable child needs a bounded ancestor. In grid/flex app layouts, the missing piece is often not another height: 100%, but min-height: 0 on the containers between the app shell and the scrollable panel.
