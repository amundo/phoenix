First Class Notes

Today was the first class. Overall, it went reasonably well.

The students seemed more engaged than before, and even the student who is usually disruptive participated, which felt like a positive sign.

Realm Creation

We were able to guide the students through creating their own small “realms,” then load those realms into the game.

Out of six student projects:

  • 5/6 loaded successfully
  • The sixth loaded, but the emojis did not match what had been edited

Even though many of the student ideas were still vague, it was fun to see the directions they wanted to go.

Emotes / Reactions

One unexpectedly successful feature was the emote system.

Students loved triggering emotes by repeatedly walking into walls just to see the reactions fire. That level of playful experimentation is a good sign.

Coordinates (X / Y)

Teaching x and y positions worked surprisingly well.

The students seemed genuinely interested in how movement maps onto coordinates. It felt like a natural introduction to graph concepts through gameplay.

QR Code Workflow

The QR code transfer system was awkward and clunky.

Issues included:

  • Slow or inconvenient scanning flow
  • Possible data corruption in at least one case
  • Friction during classroom use

This now feels like the weakest part of the current setup.

Priorities for Next Week

1. Document the Engine API

We need clear documentation for the interface that communicates with the game engine.

The long-term goal is to run engines server-side and transmit effect objects over the network.

Examples:

  • movement requests
  • collision results
  • speech effects
  • emotes
  • inventory changes
  • state updates

This API spec is the immediate priority.

2. Server-side Engine Ports

We plan to port the engine to the server side, using the Go language.

3. Replace QR Codes with Server Push

Instead of transferring realm data via QR codes, student creations should be saved directly to a server and loaded from there.

Benefits:

  • Faster classroom workflow
  • Fewer transfer errors
  • Easier persistence
  • Easier sharing
  • Better multiplayer path later

Overall Impression

The strongest signal was that the students were engaged not only by “playing,” but by experimenting with systems:

  • movement
  • coordinates
  • reactions
  • customization
  • creating worlds

That suggests the project has real educational potential if we reduce the workflow friction.
